The memory firewall

Nothing enters your agent's context without provenance that still holds. The unique lever — possible only for a coding tool because the repo is ground truth — is tying memory validity to source-file content hashes.

Verified Recall — four states, one firewall

Every memory is classified against the live repo before it can reach a model. memory_resume, the /recall prompt, the Claude Code SessionStart hook, and the proxy all run with the firewall on. It drops stale before injecting. It does not drop unsourced — unsourced means unverified, not dangerous, so it stays available for explicit lookups.

●

verified

A captured source-file hash still matches the file on disk. Code-backed and current.

●

sourced_unverified

Has a source (a command, or files present but not hashable) but no content hash to re-check. Allowed, not content-verified.

●

stale → dropped

A referenced file was deleted, or its content changed since capture. Never injected into the model.

The shareable artifact: memwarden doctor runs the same check as a report, plus conservative subject/value conflict detection — no LLM, no fuzzy black box. Run it against your current store and see how much is still earned.

Déjà Fix — never solve the same error twice

memwarden is the one process that sees every agent's sessions, so it can do what no per-tool memory can: when any agent resolves an error, it captures {error signature → root cause + fix} with provenance file-hashes. When any agent later hits a matching error, the verified fix is surfaced automatically — but only if its files still hash-match. A stale fix is never surfaced.

# Codex solved a failing test yesterday. Today Claude Code hits the same failure:
Déjà Fix (memwarden): this error was solved by codex on 2026-06-09
and the fix is verified current against your working tree.
  root cause: clock skew   ·   fix: mock NTP in conftest

# scriptable too — pipe a failing command's output straight in:
npm test 2>&1 | node dist/cli/bin.js dejafix lookup

Cross-agent (a fix learned in Codex helps Claude Code), project-scoped (a fix never leaks across repos), and safe by construction — it reuses Verified Recall, so file drift or deletion auto-suppresses it. Only verified current fixes auto-inject; sourced-but-unverified ones stay available on demand, never silently pushed.

Why file hashes are the moat

Other tools have no way to check whether a memory is still true, so they don't try. A coding agent does: the repository on disk is the source of truth. When memwarden captures a code-backed memory it records a SHA-256 content hash for each referenced file (best-effort, files up to ~2 MB). On recall it re-hashes the live file and compares. If the file moved or vanished, the memory is provably stale — not by heuristic, by hash. Nobody else ties memory validity to source-file hashes.

🔗

Tamper-evident, honestly

Every write lands in an append-only SHA-256 hash-chained oplog. memory_verify recomputes the chain, so an edit or reorder of any past entry breaks it. Tamper-evident, not tamper-proof: there is no signing, and the chain does not detect tail-truncation (dropping the newest entries).

How memory crosses your tools

Cross-tool reach is table stakes — the trust layer is the point. Still, the mechanics matter, so here they are honestly. There are exactly three ways memory reaches a tool, and up wires whichever ones each tool supports. We do not pretend any of it is magic.

Door 1 · mechanical

Hooks

Claude Code injects this project's verified memory before you type, and captures your work as it happens. The agent cannot forget to do it.

guaranteed
Door 2 · standing instruction

MCP + AGENTS.md

For Codex, Cursor, Kiro, Antigravity, OpenCode and OpenClaw, up writes an instruction telling the agent to recall first and save as it goes. This is how every cross-tool memory layer does these tools, plus /recall when you want to force it.

agent follows it
Door 3 · mechanical

The proxy

Point a tool's model base URL at the memwarden proxy on :3113 and every turn is recalled and captured with no agent cooperation. Works only where you control the endpoint: OpenCode, OpenClaw, Ollama, LM Studio, custom OpenAI base URLs.

guaranteed

Only hooks and the proxy are mechanically guaranteed. The proxy applies to model endpoints you control, not to tools with their own protocol or backend (Claude Code is covered by hooks; Cursor, Kiro and Antigravity by the standing instruction). For MCP-only tools, "automatic" means a standing instruction the agent follows, the same mechanism every memory layer uses, backed by /recall on demand.

How it works

Capture, compress on-device, store with integrity, recall the right slice under a token budget. Scoping is by canonical path, so recall never silently misses.

your tool  ──observe + hash files──▶  compress  ──▶  libSQL store + hash-chained oplog

  MiniLM embed  ──▶  TurboQuant 4-bit codes  ──▶  vector index

your tool  ◀──verified recall──  BM25 + vector (RRF), firewalled, packed under a token budget

Quick start

Two runtime dependencies. No API key. No cloud. Embeddings run on your machine.

npm install && npm run build
node dist/cli/bin.js up

# restart a tool, then in any of them:
/recall                                # or just ask "what were we working on here?"

# audit any memory store — the shareable red/yellow/green artifact:
node dist/cli/bin.js doctor .
