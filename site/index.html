The problem isn't forgetting. It's confidently wrong recall.

A stored fact silently goes stale — it points at code you've since changed or deleted — and the agent injects it with full confidence anyway. The industry has started naming this class of risk: OWASP added Memory Poisoning (ASI06) to its 2026 Agentic Top 10. Most memory still stores everything and trusts everything. A memory that points at code you deleted is worse than none. memwarden flips the default: memory is untrusted until its source still checks out.

The memory firewall

Memory whose provenance stopped checking out is dropped before it reaches the model; what passes is labeled verified / sourced / unsourced, never laundered into one pile. The unique lever — possible only for a coding tool because the repo is ground truth — is tying memory validity to source-file content hashes.

Verified Recall — four states, one firewall

Every memory is classified against the live repo before it can reach a model. memory_resume, the /recall prompt, the Claude Code SessionStart hook, and the proxy all run with the firewall on. It drops stale before injecting. It does not drop unsourced — unsourced means unverified, not dangerous, so it stays available for explicit lookups. What passes is labeled for exactly what it is — never laundered into one pile.

●

verified

A captured source-file hash still matches the file on disk. Code-backed and current. Auto-injected.

●

sourced

Has a source (a command, or files present but not hashable) but no content hash to re-check. Allowed, labeled, not content-verified.

●

stale → dropped

A referenced file was deleted, or its content changed since capture. Never injected into the model.

●

unsourced

No evidence trail. Kept for explicit lookups, labeled so the agent knows it is unverified — never silently promoted.

The shareable artifact: memwarden doctor runs the same check as a report, plus conservative subject/value conflict detection — no LLM, no fuzzy black box. memwarden why <id> explains one memory; doctor --fix-stale forgets the whole stale inbox. And when the firewall refuses something at SessionStart, the agent sees the refusal evidence — not a silent empty inject.

$ memwarden doctor .

  VERIFIED:        8 memories (code-backed, current)
  SOURCED:         3 memories (sourced, not content-verified)
  STALE:           2 memories reference files that changed/deleted
  UNSOURCED:       1 memory has no evidence
  CONFLICTS:       1 possible contradiction

  [stale]     Edit — references files that no longer match (changed: src/legacy.ts)
  [conflict] Edit may contradict Edit — same subject "auth" has incompatible values

The foreign-store audit also emits an evidence-backed action plan in --json and --html: quarantine missing-file memory, refresh drifted code facts, promote PRESENT facts into hash-verified recall, anchor free-floating memory, and wire live recall through the firewall. It recommends; it does not mutate your store.

Déjà Fix — never solve the same error twice

memwarden is the one process that sees every agent's sessions, so it can do what no per-tool memory can: when any agent resolves an error, it captures {error signature → root cause + fix} with provenance file-hashes. When any agent later hits a matching error, the verified fix is surfaced automatically — but only if its files still hash-match. A stale fix is never surfaced.

# Codex solved a failing test yesterday. Today Claude Code hits the same failure:
Déjà Fix (memwarden): this error was solved by codex on 2026-06-09
and the fix is verified current against your working tree.
  root cause: clock skew   ·   fix: mock NTP in conftest

# scriptable too — pipe a failing command's output straight in:
npm test 2>&1 | npx memwarden dejafix lookup

Cross-agent (a fix learned in Codex helps Claude Code), project-scoped (a fix never leaks across repos), and safe by construction — it reuses Verified Recall, so file drift or deletion auto-suppresses it. Only verified current fixes auto-inject; sourced-but-unverified ones stay available on demand, never silently pushed.

Source-file hashes: the ground truth

A coding agent has something general-purpose memory doesn't: the repository on disk is the source of truth. When memwarden captures a code-backed memory it records a SHA-256 content hash for each referenced file (best-effort, files up to ~2 MB). On recall it re-hashes the live file and compares. If the file moved or vanished, the memory is provably stale — not by heuristic, by hash. That's what lets the repo tell us, on every recall, whether a memory is still earned.

🔗

Tamper-evident, honestly

Every write lands in an append-only SHA-256 hash-chained oplog. memory_verify recomputes the chain, so an edit or reorder of any past entry breaks it. Tamper-evident, not tamper-proof: there is no signing, and the chain does not detect tail-truncation (dropping the newest entries).

🧾

Erasure with receipts

memwarden forget <id> returns an offline-checkable delete receipt (create + delete entry hashes, chain head). forget --erase nulls payloads and re-chains; memwarden compact reclaims bytes so content actually leaves the file. Receipts carry a real contentErased boolean — no silent lies about deletion.

How memory crosses your tools

Cross-tool reach is table stakes — the trust layer is the point. Still, the mechanics matter, so here they are honestly. There are exactly three ways memory reaches a tool, and up wires whichever ones each tool supports. We do not pretend any of it is magic.

Door 1 · mechanical

Native hooks, everywhere

Claude Code, Codex, Cursor, Gemini CLI, Kiro and OpenCode all get real lifecycle hooks in their own config format: a session-start hook injects this project's verified memory before you type, and capture happens as tools run. The agent cannot forget to do it — there is no instruction to follow. One canonical event layer speaks each host's dialect (memwarden hook <event> --host cursor, etc.).

guaranteed
Door 2 · mechanical

The proxy

Point a tool's model base URL at the memwarden proxy on :3141 and every turn is recalled and captured with no agent cooperation. On a secured install the tool's API key is your memwarden secret — the proxy refuses other local processes, so nothing else can spend your upstream key or poison capture. Works where you control the endpoint: Ollama, LM Studio, custom OpenAI base URLs.

guaranteed
Door 3 · fallback

MCP + AGENTS.md

Tools without a hook surface (today: OpenClaw) still get the MCP server, plus an AGENTS.md instruction to recall first and save as they go — explicitly the fallback, not the mechanism, and /recall forces it on demand.

agent follows it

Honest small print: Codex requires one-time hook trust-pinning (run /hooks inside Codex); Kiro hooks attach per custom agent; OpenCode capture is mechanical while injection is best-effort. memwarden status shows exactly which of these are live rather than assumed — that column is the difference between "configured" and "working".

How it works

Capture, compress on-device, store with integrity, recall the right slice under a token budget. Scoping is by canonical path, so recall never silently misses.

At this scale pure 4-bit codes match full precision on every metric at 5.9× smaller vectors; at 10,000 distractors they give up ~7 points of R@10 unless exact rescoring is enabled (which keeps full vectors resident) — run the benchmark with --distractors 10000 to see both sides.

Quick start

Two runtime dependencies. No API key. No cloud. Embeddings run on your machine.

# start here — audit the memory you ALREADY have, zero install, read-only:
npx memwarden audit ~/.claude-mem/claude-mem.db --root ~/code/my-repo
npx memwarden audit CLAUDE.md             # or any markdown pile / Mem0 JSON export

# then wire everything — one stable install:
npm install -g memwarden && memwarden up

# restart a tool, then in any of them:
/recall                                # or just ask "what were we working on here?"

# audit your own brain (verified / sourced / stale / unsourced + footprint):
npx memwarden doctor .

# delete with a tamper-evident receipt (contentErased:false until --erase):
npx memwarden forget <observationId>
npx memwarden forget <observationId> --erase   # null payloads + re-chain
npx memwarden compact                          # reclaim bytes; content leaves the file
